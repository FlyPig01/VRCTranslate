# VRCTranslate OCR 模型升级方案

## 1. 方案目标

在不引入 PyTorch、PaddlePaddle 等大型运行环境的前提下，提高 VRChat 中文和日文聊天文字的识别准确率。

本方案继续采用 PaddleOCR/RapidOCR 与 ONNX Runtime，只支持：

- 中文识别包。
- 日文识别包。

不提供韩文、拉丁语言、西里尔语言、阿拉伯语言或其他语言包，也不设计通用语言模型下载入口。

## 2. 当前实现

当前项目使用 `rapidocr-onnxruntime 1.4.4`，模型组合如下：

| 功能 | 当前模型 | 大小 |
| --- | --- | ---: |
| 文字检测 | PP-OCRv4 Mobile | 约 4.53MB |
| 中文识别 | PP-OCRv4 Mobile | 约 10.35MB |
| 日文识别 | Japan PP-OCRv4 Mobile | 约 9.30MB |
| 文字方向判断 | PP-OCR Mobile v2 | 约 0.56MB |
| 合计 |  | 约 24.74MB |

当前模型侧重低存储、低延迟，适合清晰且对比度较高的文字。面对 VRChat 的小字号、半透明背景、描边、阴影、发光和动态画面时，识别准确率有限。

当前默认识别置信度为 `0.8`，部分正确但置信度较低的结果可能在应用层被过滤。当前“自动语言”也不是真正的中日文自动判断，而是直接使用日文识别模型。

## 3. 推荐模型组合

采用“轻量检测 + 高精度识别”的组合：

| 功能 | 推荐模型级别 | 说明 |
| --- | --- | --- |
| 文字检测 | PP-OCRv5 Mobile Det | 处理完整选区，优先控制 CPU 占用 |
| 中文识别 | PP-OCRv5 高精度中文 Rec | 优先使用兼容的 Server 或高精度 ONNX 模型 |
| 日文识别 | Japan PP-OCRv5 专用 Rec | 必须使用日文专用字符集和模型 |
| 方向判断 | 继续保留或按测试关闭 | VRChat Chatbox 通常为水平文字 |

`Server` 表示更大的本地模型，不代表需要连接服务器。

不建议默认使用 PP-OCRv5 Server Det。检测模型需要处理整个截图选区，换成 Server 版后可能明显增加 CPU 占用和单帧耗时。高精度识别模型只处理已经裁剪出的文字区域，性能成本相对更可控。

## 4. 模型包范围

只定义两个识别包：

### 4.1 中文识别包

用于：

- 简体中文。
- 常见英文、数字和符号。
- VRChat 中以中文为主的聊天内容。

中文识别包不负责完整日文假名识别。

### 4.2 日文识别包

用于：

- 平假名。
- 片假名。
- 日文常用汉字。
- 常见英文、数字和符号。
- VRChat 中以日文为主的聊天内容。

日文识别包必须使用模型对应的日文字符字典，不能只把中文模型的字符字典替换为日文字典。

## 5. 便携目录

模型只保存到软件目录：

```text
VRCTranslate/
├── VRCTranslate.exe
├── _internal/
└── data/
    └── models/
        └── ocr/
            ├── manifest.json
            ├── shared/
            │   ├── detection.onnx
            │   └── orientation.onnx
            ├── zh/
            │   ├── recognition.onnx
            │   ├── characters.txt
            │   └── metadata.json
            └── ja/
                ├── recognition.onnx
                ├── characters.txt
                └── metadata.json
```

不得使用 AppData、用户目录或 Hugging Face 等第三方默认缓存目录。

应用不保存 OCR 截图。模型下载临时文件只能写入软件目录下的 `data/cache`，安装完成后删除。

## 6. 模型清单

`manifest.json` 至少记录：

- 模型包编号。
- 中文或日文语言代码。
- 模型版本。
- 下载地址。
- 下载大小。
- 安装后大小。
- SHA-256 校验值。
- 对应字符字典。
- 模型输入尺寸。
- RapidOCR 解码参数。
- 是否已经安装。

下载时先写入临时文件，校验 SHA-256 成功后再原子替换正式模型。下载失败或校验失败时不得破坏当前可用模型。

## 7. 语言选择规则

OCR 路由只提供以下识别语言：

- 中文。
- 日文。

不再提供含义不明确的“自动识别”作为 OCR 模型选择。如果需要保留旧配置兼容：

- 旧配置为中文时选择中文包。
- 旧配置为日文时选择日文包。
- 旧配置为自动识别时回退到日文包，并在设置页要求用户明确确认中文或日文。

同一 OCR 会话只加载一个识别包，不同时运行中、日文两个识别器，避免 CPU 和识别耗时翻倍。

## 8. 参数调整

模型升级时同步评估：

- 默认识别置信度由 `0.8` 调整到约 `0.68`。
- 保留可配置置信度，建议范围为 `0.60～0.85`。
- 继续使用精确的小范围 OCR 选区。
- 重新评估当前 1.5 倍放大和锐化预处理。
- 测试关闭水平文字不需要的方向分类器。
- 限制 ONNX Runtime CPU 线程，避免影响 VRChat 帧率。

不能只通过降低置信度追求识别数量。最终阈值必须同时比较漏识别和误识别。

## 9. 存储预算

预计模型占用：

| 内容 | 预计占用 |
| --- | ---: |
| 共享 Mobile 检测与方向模型 | 约 5～15MB |
| 中文高精度识别包 | 约 50～120MB |
| 日文专用识别包 | 约 10～80MB |
| 总计 | 约 65～215MB |

具体大小以最终选择并转换完成的 ONNX 文件为准。

OpenCV、NumPy 和 ONNX Runtime 已经是现有 OCR 依赖，更换模型不会重复增加这些运行环境。

## 10. 性能策略

性能优先级如下：

1. 保持 Mobile 检测模型。
2. 只升级识别模型。
3. 同一时间只加载一个语言包。
4. OCR 识别任务保持单实例，不允许同一帧并行运行多个模型。
5. 新帧到达时不得无限堆积识别任务。
6. OCR 与翻译结果继续遵守现有任务有效期。

如果升级后仍然完全检测不到文字，再单独测试 Server Det，不直接把 Server Det 设为默认模型。

## 11. RapidOCR 兼容性

当前 `rapidocr-onnxruntime 1.4.4` 不保证直接兼容所有 PP-OCRv5 模型。实施前必须验证：

- ONNX 输入名称与尺寸。
- 输出结构。
- CTC 解码方式。
- 字符字典顺序。
- 检测后处理参数。
- 模型是否携带字符元数据。

如果当前运行库不能正确加载 PP-OCRv5，应迁移到支持对应模型的新版 RapidOCR，但仍使用 ONNX Runtime，不能引入 PyTorch 或完整 PaddlePaddle。

## 12. 实施顺序

### 第一阶段：基准测试

- 记录当前 PP-OCRv4 的单帧耗时、识别结果和置信度。
- 分别测试中文和日文 VRChat 文字。
- 测试只在内存中完成，不保存截图。

### 第二阶段：识别模型升级

- 保留当前 Mobile Det。
- 接入中文高精度识别模型。
- 接入日文专用识别模型。
- 验证字符字典和解码结果。

### 第三阶段：检测模型升级

- 把 Mobile Det 升级到 PP-OCRv5 Mobile Det。
- 比较漏检率和单帧耗时。
- 只有 Mobile Det 仍明显漏检时，才评估 Server Det。

### 第四阶段：语言包管理

- 建立中文、日文模型清单。
- 实现下载、校验、安装和删除。
- 确保所有文件只写入软件 `data` 目录。
- 缺少模型时显示明确说明，不静默下载。

## 13. 验收标准

### 功能

- 中文路由只加载中文包。
- 日文路由只加载日文包。
- 缺少模型时不会启动错误的识别器。
- 模型切换后旧识别实例被安全释放。
- 不生成或保存任何截图。

### 质量

- 中文小字号识别优于当前 PP-OCRv4 Mobile。
- 平假名、片假名和相似字符识别优于当前日文模型。
- 不依赖大量硬编码字符替换掩盖模型错误。
- 动态背景下的漏识别率和误识别率均有记录。

### 性能

- 单次 OCR 耗时低于当前 OCR 扫描间隔。
- 连续运行时不出现识别任务堆积。
- OCR 运行时 VRChat 不出现明显持续掉帧。
- 切换语言后内存中只保留当前识别模型。

### 存储与隐私

- 只存在中文和日文两个可下载识别包。
- 模型与临时文件全部位于软件目录。
- 不使用 C 盘用户缓存作为回退。
- 不保存截图、识别历史或聊天数据库。

## 14. 最终建议

默认采用：

```text
PP-OCRv5 Mobile Det
+ 中文高精度 Rec
+ Japan PP-OCRv5 专用 Rec
```

优先升级识别模型，暂不使用 Server Det。完成中文和日文实测后，再决定是否需要更大的检测模型。
