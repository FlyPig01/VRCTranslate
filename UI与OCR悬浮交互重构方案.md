# VRCTranslate UI 与 OCR 悬浮交互重构方案

> 文档状态：已审批并实施
> 编写日期：2026-07-17
> 适用范围：Windows 10/11、Python 3.11.4、VRChat PC 桌面模式
> 当前阶段：功能、配置迁移、样式和自动化测试已完成；未进行打包

> 2026-07-18 界面复审修订：快捷输入页与 OCR 功能页改为自动保存，不再显示保存/放弃状态条；删除“显示快捷输入浮窗”和“显示 OCR 悬浮球”按钮；OCR 页恢复目标程序选择；两个功能页均固定提供最近一次译文卡片。本修订优先于下文早期的手动保存和隐藏目标选择设计，设置主页面仍保留原有手动保存机制。

## 1. 方案结论

本次重构将“常用功能设置”放回对应功能页面，并用一个 OCR 识别框和一个 OCR 悬浮球替代主页面上分散的窗口选择、区域选择和启停控件。

项目现有的 `domain / application / infrastructure / presentation` 分层总体合理，不需要推倒重建。主要问题集中在 Qt 展示层：页面直接承担了过多状态、OCR 控制器依赖页面控件取值、设置只使用一个整页草稿、悬浮窗口职责不够细。方案将在现有分层内拆分这些职责，不改变核心翻译和 OSC 架构。

本次继续遵守以下项目约束：

- 只支持 PC 桌面模式，不加入 PCVR 交互。
- 配置仍只写入软件目录下的 `data/config.json`，不拆数据库，不回退到 C 盘用户目录。
- 截图只在内存中处理，不保存截图、聊天历史或 OCR 历史。
- 不恢复 Argos Translate，也不加入新的离线翻译服务。
- OCR 模型升级沿用独立的《OCR模型升级方案.md》，本次不混入模型下载与切换功能。
- 本轮不打包；打包仍是功能和测试全部完成后的最后步骤。

| 页面 | 当前主要内容 | 主要问题 |

## 4. 快捷输入页面设计

快捷输入页面不增加与当前任务无关的复杂功能，建议布局如下：

```text
快捷输入                                      已保存  [放弃] [保存修改]
────────────────────────────────────────────────────────────
快捷输入浮窗
  当前翻译档案 / 运行状态 / [显示快捷输入浮窗]

最近一次翻译
  原文
  译文

浮窗设置
  [✓] 快捷输入浮窗置顶
  浮窗宽度  [ 数值输入 ]
```

行为约定：

- 修改置顶或宽度时立即在输入浮窗中预览，但尚不写入配置。
- 点击“保存修改”后才写入 `data/config.json`。
- 点击“放弃”恢复到本页上一次保存值，并同步恢复浮窗预览。
- 本轮不擅自增加输入浮窗位置输入框；位置仍由拖动决定。

OSC 设置页删除“快捷输入浮窗”整张卡片，只保留 OSC 连接和发送行为。

## 5. OCR 主页面设计

OCR 主页面删除当前的目标窗口下拉框、刷新按钮、当前选区、选择识别区域、启动/停止 OCR 按钮和会话诊断表格。建议改为以下结构：

```text
OCR 翻译                                      已保存  [放弃] [保存修改]
────────────────────────────────────────────────────────────
OCR 工作状态
  状态：未运行 / 单次识别中 / 持续识别中 / 等待画面变化 / 错误
  目标：VRChat（未找到时给出明确提示）
  模式：单次 / 持续
  语言：日文 → 中文       翻译服务：当前 OCR 路由档案
  [显示悬浮球]

OCR 识别工具
  [✓] 悬浮球置顶
  说明：选区、模式和启停均通过悬浮球及识别框操作

OCR 译文浮窗
  [✓] 置顶    [ ] 鼠标穿透    [✓] 显示原文
  背景透明度 / 字号 / 最大条数 / 显示时长
  当前位置与尺寸摘要
  样式预览
  [显示译文浮窗] [重置位置和尺寸]
```

保存条固定在页面标题区域或固定底栏，不能跟随内容滚动离开视野。常用状态和“显示悬浮球”必须在无需滚动的位置。

状态、模式、语言和翻译服务档案在 OCR 主页面中全部是只读摘要，不提供下拉框、单选按钮或直接编辑入口。识别模式由识别框或悬浮球切换；语言和翻译档案仍在“翻译模型配置 → OCR 翻译路由”中修改。

## 6. OCR 识别区域框

### 6.1 外观

识别框是独立的无系统标题栏工具窗口，视觉语言与 OCR 译文浮窗统一，但用途不同：

- 内部透明，不覆盖 VRChat 画面。
- 使用“深色外描边 + 高亮内描边”的双层边框，在白色、黑色和彩色场景下都能看清。
- 顶部控制条位于实际捕获矩形之外，显示目标、模式和运行状态。
- 控制条提供“单次 / 持续”两项模式选择和右上角 `X`。
- 四边和四角提供清晰但不过度突出的缩放热区。
- 运行时允许点击穿透透明内容区，不妨碍操作 VRChat；控制条和缩放热区仍可交互。

### 6.2 几何和关闭语义

- 拖动控制条移动识别区域，拖动边角改变识别区域尺寸。
- 识别区域始终限制在所选目标窗口的客户区内。
- 目标窗口移动或缩放时，识别框随客户区更新，不把屏幕绝对坐标误当成客户区坐标。
- 点击识别框 `X`：停止当前 OCR、清除当前区域并关闭识别框，但保留悬浮球，方便重新框选。
- 悬浮球菜单中的“退出 OCR 工具”：停止 OCR，同时关闭识别框和悬浮球。
- 主页面中的“显示悬浮球”用于恢复被隐藏或关闭的悬浮球。

### 6.3 拖动时的性能策略

- 窗口移动和绘制只在 Qt UI 线程执行，不在移动事件中调用 OCR、网络翻译或磁盘保存。
- 拖动或缩放期间暂停发起新一帧识别，释放鼠标后再用最终区域恢复，避免识别任务与几何更新争用。
- 几何变化只在拖动结束后更新页面草稿，不按每个鼠标移动事件写配置。
- OCR 结果到达时先进入内存缓冲；若用户正在拖动译文浮窗或识别框，只保留必要的最新 UI 更新，拖动结束后一次刷新。
- 状态文本刷新限频，避免高频信号造成布局重算和拖动卡顿。

## 7. OCR 悬浮球

### 7.1 基本交互

| 操作 | 行为 |
| --- | --- |
| 单击 | 已有有效选区时开始或暂停当前模式；没有选区时直接进入框选 |
| 拖动 | 移动悬浮球，不触发启停 |
| 右键或展开菜单 | 打开完整 OCR 操作菜单 |
| 状态颜色 | 空闲、运行、等待变化、错误使用不同但可辨认的颜色和图标 |

菜单包含：

- 选择或重新选择目标窗口。
- 框选或重新框选识别区域。
- 单次识别。
- 单次识别。
- 持续识别（周期取帧并判断局部画面变化）。
- 显示或隐藏识别框。
- 暂停或继续 OCR。
- 退出 OCR 工具。

### 7.2 目标窗口选择

- 第一次使用时优先查找 VRChat 窗口，但不静默选择其他覆盖层窗口。
- 找不到 VRChat 时由悬浮球弹出精简窗口列表，让玩家明确选择。
- 配置只持久化可再次匹配的窗口标题信息，不持久化重启后无效的 HWND。
- 进入框选时将目标窗口激活，并把选择层限制在目标客户区；不通过反复隐藏和重新显示主窗口制造闪烁。

### 7.3 置顶规则

- “悬浮球置顶”是用户可保存的设置，放在 OCR 主页面的“OCR 识别工具”卡片中。
- 关闭置顶后，悬浮球是普通工具窗口；主页面始终保留“显示悬浮球”恢复入口。
- 识别框不做无条件的全局置顶，它只需要跟随并显示在目标 VRChat 窗口之上，避免切换到其他应用后仍遮挡桌面。

### 7.4 悬浮球图片设计

悬浮球使用项目内 SVG 矢量资源，不通过 Python 临时绘制，也不依赖网络图片。设计语言与现有应用图标保持一致：

- 主轮廓为 48×48 的深蓝圆形，外侧使用半透明深色描边，适配明暗背景。
- 中心图形使用“四角取景框 + 文字符号”，表达 OCR 取景和文字识别，而不是使用容易与录音混淆的麦克风图标。
- 图形只使用少量纯色和简单路径，缩放到 32×32 时仍然清楚。
- 透明背景，悬浮球窗口本身不绘制额外矩形底色。

计划新增以下资源：

| 文件 | 状态颜色 | 用途 |
| --- | --- | --- |
| `ocr_orb_idle.svg` | 蓝色 | 已就绪或暂停 |
| `ocr_orb_running.svg` | 绿色 | 正在持续识别 |
| `ocr_orb_waiting.svg` | 金色 | 正在等待画面达到变化阈值 |
| `ocr_orb_error.svg` | 红色 | 目标失效、捕获或 OCR 错误 |

状态切换只替换已经缓存的图标，不在每一帧重新生成图片，避免增加 UI 开销。

## 8. 两种 OCR 识别模式

新增明确的 `recognition_mode`。周期取帧和画面变化判断合并为一个“持续识别”模式，变化阈值成为持续模式内部的处理参数，而不是第三种模式。

| 模式 | 启动后行为 | 停止条件 | 适用场景 |
| --- | --- | --- | --- |
| `single` 单次识别 | 捕获、OCR、翻译一次 | 本轮任务提交完成后自动回到空闲；识别框保留 | 玩家只想翻译当前一句文字 |
| `continuous` 持续识别 | 按 `interval_ms` 取帧；只对达到变化阈值的画面进入 OCR，再按文字框位置筛选新增或改变的文字 | 用户暂停、关闭、目标失效或发生错误 | 同一识别框内同时存在静止文字和持续变化文字 |

状态流转：

```text
未选择区域 ──框选──> 空闲
空闲 ──单次──> 捕获/识别/翻译 ──完成──> 空闲
空闲 ──持续──> 周期取帧/等待变化 ──局部变化──> 识别 ──完成──> 周期取帧/等待变化
任意运行状态 ──目标失效/错误──> 已暂停并提示
任意状态 ──识别框 X──> 停止并清除区域
```

并发和重复处理规则：

- 同一个 OCR 会话只允许一个捕获/OCR 帧处理任务运行，不并发识别多帧。
- 在线翻译继续使用有界队列和结果有效期，过期结果不更新浮窗。
- 删除用户可配置的“重复冷却秒数”和 `TextDeduplicator` 的时间窗口。
- 持续模式保留上一轮 OCR 文字框快照，按文字内容和空间重叠判断每个区域是否改变。静止区域不会因其他区域变化而被重复翻译；同一位置出现新文字时会正常提交。
- 保留非时间型的在途任务合并：同一位置、同一文本已有翻译任务时不重复提交；这用于防止队列膨胀，不作为玩家可调的“重复冷却”。
- 切换模式、重新框选或更换目标窗口时重置画面基线和会话内在途状态。

## 9. 保存状态与草稿机制

### 9.1 页面表现

每个可编辑页面都固定显示以下状态之一：

- `已保存`：当前页面控件与持久化配置一致。
- `有未保存修改`：至少一个由本页负责的字段发生变化。
- `保存中…`：正在原子写入配置。
- `保存失败`：保留草稿并显示可理解的错误，不丢弃玩家输入。

固定操作为：

- `保存修改`：只合并并保存本页负责的字段。
- `放弃修改`：恢复本页基线，并撤销尚未保存的实时预览。

### 9.2 分区草稿

仍然只有一个 `data/config.json`，但 UI 使用按职责分区的草稿：

| 草稿范围 | 字段所有者 |
| --- | --- |
| 快捷输入 | `ui.input_topmost`、`ui.input_width`，以及拖动后需要保存的输入浮窗位置 |
| OCR 展示与工具 | 译文浮窗字段、悬浮球字段、最后使用的 OCR 模式 |
| 翻译模型 | `translation.profiles`、`translation.self_route`、`translation.ocr_route` |
| OSC | `osc.*` |
| OCR 引擎 | 捕获后端、间隔、置信度、变化阈值、队列上限和结果有效期 |

保存某一页时，必须基于最新的当前配置只覆盖该页字段，不能用整份旧草稿覆盖其他页面已保存的修改。

### 9.3 离开和退出

- 从一个存在未保存修改的页面切换到其他主页面时，弹出“保存 / 放弃 / 取消切换”。
- 设置分区之间切换时，允许保留多个分区草稿，但分区导航要显示未保存圆点。
- 关闭程序时统一列出存在未保存修改的页面，再选择保存全部、放弃全部或取消退出。
- 只读的“便携数据与诊断”页面不伪造保存按钮，明确显示本页没有待保存设置。

## 10. 配置字段和迁移

本次涉及配置语义变化，建议将配置版本由 v2 升级为 v3，并建立明确的 v2 → v3 迁移函数，不能继续让所有旧版本都走当前的 v1 迁移逻辑。

### 10.1 字段变化

| 配置位置 | 操作 | 说明 |
| --- | --- | --- |
| `ocr.recognition_mode` | 新增 | 取值为 `single`、`continuous` |
| `ocr.duplicate_seconds` | 删除 | 不再读取、不再显示，保存 v3 时移除 |
| `ui.ocr_orb_topmost` | 新增 | 悬浮球是否置顶，默认开启 |
| `ui.ocr_orb_x` / `ui.ocr_orb_y` | 新增 | 悬浮球位置，负值表示自动放置 |
| 现有 `ocr.region_*` | 保留 | 继续表示目标窗口客户区内的物理像素区域 |

| 现有 `ui.input_*` | 保留 | 仅改变编辑入口，不改变语义 |

### 10.2 v2 → v3 迁移规则

- 原 v2 配置统一迁移为 `continuous`，保留原 `interval_ms` 和 `change_threshold`。
- 删除 `duplicate_seconds`，不把旧秒数映射为隐藏配置。
- 新增的悬浮球位置使用自动放置，置顶默认开启。
- 保留原目标窗口标题、客户区选区、输入浮窗和译文浮窗几何。
- 迁移前按现有便携数据规则备份旧配置；迁移失败时不覆盖原文件。

根目录遗留的 `config.json` 与实际便携数据 `data/config.json` 不同时作为运行配置源。本次继续以路径管理器解析出的 `data/config.json` 为准，不新增第二套保存机制。

## 12. 目录结构与职责调整

### 12.1 合理性判断

现有四层结构是合理的，应继续保留：


## 13. 控制器和状态边界

重构后只有一份 `OcrRuntimeState` 作为 OCR UI 的运行状态来源，至少包含：

- 当前目标窗口信息。
- 客户区识别区域。
- 当前识别模式。
- 空闲、运行、等待变化、暂停、停止中、错误等状态。
- 识别框和悬浮球可见性。

职责约定：

- 页面只展示状态和编辑草稿，不再通过公开的下拉框让控制器自行读取值。
- 悬浮球和识别框只发出用户意图，例如“重新框选”“切换模式”“开始”“暂停”。
- `OcrController` 负责协调子控制器，不负责绘制窗口，也不直接拼装页面控件。
- `OcrTargetController` 接收明确的目标对象，不再依赖 `OcrPage.window_combo`。
- `OcrSessionController` 根据模式决定执行一次或循环，不负责译文浮窗位置。
- 保存控制器只提交所属字段，不从各页面收集一整份可能过期的 `AppSettings`。

## 14. DPI、多屏和捕获排除

### 14.1 坐标规则

- 配置中的 OCR 选区统一为 Win32 目标窗口客户区物理像素坐标。
- Qt 鼠标坐标只存在于展示层；提交选区前按“目标客户区物理尺寸 ÷ 选择层实际尺寸”分别换算 X/Y。
- 每次框选和目标窗口跨屏后重新读取目标所在屏幕的 DPI，不能只使用主屏缩放率。
- 支持 100%、125%、150%、175%、200% 缩放，以及主副屏缩放不一致和副屏负坐标。
- 目标窗口尺寸变化后对区域做边界收敛；区域失效时暂停并要求重新框选，不能继续捕获错误坐标。

### 14.2 排除自身窗口

建立统一的捕获排除注册，覆盖：

- 主窗口。
- 快捷输入浮窗。
- OCR 译文浮窗。
- OCR 识别框。
- OCR 悬浮球。
- 临时区域选择层。

优先使用 Windows 的窗口捕获排除能力。识别框的控制条和边框还应绘制在实际捕获矩形之外，使捕获矩形即使在后端不支持排除时也不会包含识别框装饰。

若当前系统或捕获后端确实无法排除自身窗口，应暂停并给出明确提示，不采用每帧隐藏/显示窗口的方式，因为该方式会造成闪烁、焦点抢占和莫名弹窗。整个流程仍不得把捕获帧写入磁盘。

### 14.3 目标窗口状态

- VRChat 最小化或客户区不可用时暂停捕获，悬浮球显示暂停状态。
- VRChat 恢复后重新校验客户区和 DPI，再恢复当前模式。
- VRChat 关闭后停止会话，但保留悬浮球和清晰提示，方便重新选择窗口。
