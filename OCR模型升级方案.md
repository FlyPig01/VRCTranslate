# VRCTranslate OCR 模型升级方案

> 文档状态：已实施，自动化与本地模型初始化测试已通过，待真实 VRChat 画面验收
>
> 2026-07-18 新增需求：OCR 译文支持可选的“识别区域嵌字”，在识别框内尽量按原文位置和字号覆盖显示译文。

## 1. 方案目标

在不引入 PyTorch、PaddlePaddle 等大型运行环境的前提下，提高 VRChat 中文和日文聊天文字的识别准确率。

同时增加可选的 OCR 译文嵌字模式：译文在 OCR 识别区域内覆盖对应原文，位置和字号尽量接近原文字体；用户仍可选择使用现有独立译文浮窗。

本方案继续采用 PaddleOCR/RapidOCR 与 ONNX Runtime，只支持：

- 中文识别包。
- 日文识别包。

不提供韩文、拉丁语言、西里尔语言、阿拉伯语言或其他语言包，也不设计通用语言模型下载入口。

## 2. 当前实现

项目已经迁移到 `rapidocr 3.9.1` 与 `onnxruntime 1.27.0`。模型不随源码自动下载，用户在“设置 → OCR → 本地 OCR 模型”中按需安装：

| 功能 | 当前模型 | 文件大小 |
| --- | --- | ---: |
| 文字检测（共享） | PP-OCRv5 Mobile Det | 4,819,576 字节 |
| 中文识别 | PP-OCRv5 Server Rec | 84,577,022 字节 |
| 日文识别 | PP-OCRv6 Medium Rec | 76,629,984 字节 |
| 文字方向模型（共享） | PP-OCRv4 Mobile | 585,532 字节 |
| 中文完整安装 | 检测 + 中文识别 + 方向 | 约 90MB（十进制） |
| 日文完整安装 | 检测 + 日文识别 + 方向 | 约 82MB（十进制） |

官方 RapidOCR 模型库没有可直接采用的日文 PP-OCRv5 专用识别模型，因此日文最终选择支持日语字符集的 PP-OCRv6 Medium，而不是使用错误字典强行适配中文模型。

默认识别置信度已调整为 `0.68`。OCR 源语言只提供中文和日文；旧配置中的“自动”会迁移为日文，不再把它作为可见选项。

## 3. 推荐模型组合

采用“轻量检测 + 高精度识别”的组合：

| 功能 | 已采用模型 | 说明 |
| --- | --- | --- |
| 文字检测 | PP-OCRv5 Mobile Det | 处理完整选区，优先控制 CPU 占用 |
| 中文识别 | PP-OCRv5 Server Rec | 提升小字号中文识别精度 |
| 日文识别 | PP-OCRv6 Medium Rec | 支持日文字符集，体积和精度较均衡 |
| 方向判断 | PP-OCRv4 Mobile | 保留模型文件；当前水平文本流程关闭方向分类调用 |

`Server` 表示更大的本地模型，不代表需要连接服务器。

不建议默认使用 PP-OCRv5 Server Det。检测模型需要处理整个截图选区，换成 Server 版后可能明显增加 CPU 占用和单帧耗时。高精度识别模型只处理已经裁剪出的文字区域，性能成本相对更可控。

## 4. 模型包范围

只定义两个识别包：

### 4.1 中文识别包

用于：

- 简体中文。
- 常见英文、数字和符号。
- VRChat 中以中文为主的聊天内容。

中文识别包不负责完整日文假名识别。

### 4.2 日文识别包

用于：

- 平假名。
- 片假名。
- 日文常用汉字。
- 常见英文、数字和符号。
- VRChat 中以日文为主的聊天内容。

日文识别包必须使用模型对应的日文字符字典，不能只把中文模型的字符字典替换为日文字典。

## 5. 便携目录

模型只保存到软件目录：

```text
VRCTranslate/
├── VRCTranslate.exe
├── _internal/
└── data/
    └── models/
        └── ocr/
            ├── manifest.json
            ├── shared/
            │   ├── detection.onnx
            │   └── orientation.onnx
            ├── zh/
            │   └── recognition.onnx
            └── ja/
                └── recognition.onnx
```

不得使用 AppData、用户目录或 Hugging Face 等第三方默认缓存目录。

应用不保存 OCR 截图。模型下载临时文件只能写入软件目录下的 `data/cache`，安装完成后删除。

## 6. 模型清单

下载所需的官方地址、文件大小、SHA-256、语言、识别版本与模型类型保存在程序内置模型清单中。ONNX 模型包含解码所需字符元数据，因此不生成独立的 `characters.txt`。

运行时 `manifest.json` 只记录清单版本、各语言包版本及安装状态，不保存 API 密钥或识别内容。内置模型清单记录：

- 模型包编号。
- 中文或日文语言代码。
- 模型版本。
- 下载地址。
- 下载大小。
- SHA-256 校验值。
- 模型在便携目录中的相对路径。
- RapidOCR 识别版本。
- 识别模型类型。

下载时先写入临时文件，校验 SHA-256 成功后再原子替换正式模型。下载失败或校验失败时不得破坏当前可用模型。

## 7. 语言选择规则

OCR 路由只提供以下识别语言：

- 中文。
- 日文。

不再提供含义不明确的“自动识别”作为 OCR 模型选择。如果需要保留旧配置兼容：

- 旧配置为中文时选择中文包。
- 旧配置为日文时选择日文包。
- 旧配置为自动识别时迁移到日文包；用户之后可以在翻译路由中改为中文。

同一 OCR 会话只加载一个识别包，不同时运行中、日文两个识别器，避免 CPU 和识别耗时翻倍。

## 8. 参数调整

模型升级时同步评估：

- 默认识别置信度由 `0.8` 调整到约 `0.68`。
- 保留可配置置信度，建议范围为 `0.60～0.85`。
- 继续使用精确的小范围 OCR 选区。
- 已取消旧版 1.5 倍外部放大，只保留不改变图像尺寸的锐化，保证模型框与捕获帧坐标一致。
- 水平文字流程关闭方向分类器调用，但保留共享方向模型文件以兼容模型初始化。
- ONNX Runtime 限制为 2 个算子内线程和 1 个算子间线程，避免影响 VRChat 帧率。

不能只通过降低置信度追求识别数量。最终阈值必须同时比较漏识别和误识别。

## 9. 存储预算

预计模型占用：

| 内容 | 实际文件大小 |
| --- | ---: |
| 共享 Mobile 检测与方向模型 | 5,405,108 字节 |
| 中文高精度识别模型 | 84,577,022 字节 |
| 日文高精度识别模型 | 76,629,984 字节 |
| 中文与日文全部安装 | 166,612,114 字节，约 166.6MB（十进制） |

OpenCV、NumPy 和 ONNX Runtime 已经是现有 OCR 依赖，更换模型不会重复增加这些运行环境。

## 10. 性能策略

性能优先级如下：

1. 保持 Mobile 检测模型。
2. 识别使用高精度模型，检测仍使用 Mobile 模型。
3. 同一时间只加载一个语言包。
4. OCR 识别任务保持单实例，不允许同一帧并行运行多个模型。
5. 新帧到达时不得无限堆积识别任务。
6. OCR 与翻译结果继续遵守现有任务有效期。

如果升级后仍然完全检测不到文字，再单独测试 Server Det，不直接把 Server Det 设为默认模型。

## 11. RapidOCR 兼容性

旧版 `rapidocr-onnxruntime 1.4.4` 不保证直接兼容所选模型，因此实现已迁移到 `rapidocr 3.9.1`，并完成以下兼容性验证：

- ONNX 输入名称与尺寸。
- 输出结构。
- CTC 解码方式。
- 字符字典顺序。
- 检测后处理参数。
- 模型是否携带字符元数据。

中文、日文模型均已完成下载校验、ONNX 初始化和空白内存帧推理。运行时仍只使用 ONNX Runtime，没有引入 PyTorch 或完整 PaddlePaddle。

## 12. 识别区域嵌字需求与可行性

### 12.1 需求定义

“嵌字”不是修改 VRChat 自身画面，也不向 VRChat 进程注入内容，而是在目标窗口上方建立一个透明、鼠标穿透的译文绘制层。

嵌字模式需要满足：

- 用户可以选择是否启用嵌字。
- 译文只显示在已经框选的 OCR 识别区域内。
- 每段译文尽量覆盖其对应的原文位置。
- 字号根据原文 OCR 文字框高度自动估算，不使用固定字号。
- 译文比原文长时允许自动换行和有限缩小，但不能绘制到识别区域之外。
- 嵌字层不能抢占键盘或鼠标输入，不能影响 VRChat 操作。
- 嵌字层不能被下一次 OCR 捕获，避免把译文再次识别并形成循环。
- 不保存截图、文字背景或嵌字历史。

### 12.2 可行性结论

| 项目 | 结论 | 原因 |
| --- | --- | --- |
| 获取原文位置 | 可行 | RapidOCR 已返回每段文字的多边形坐标，当前 `OcrText` 也已经保留 `box` |
| 在识别框内覆盖绘制 | 可行 | Qt 可以建立透明、无边框、鼠标穿透的独立绘制层，并与目标窗口位置同步 |
| 字号接近原文 | 基本可行 | 可由文字框高度估算字号，再使用字体度量进行二次适配 |
| 位置接近原文 | 基本可行 | 可将 OCR 坐标映射到识别区域和屏幕坐标，但必须处理预处理缩放与 Windows DPI |
| 完全复刻原字体 | 不可保证 | OCR 不提供字体名称、字重、字距、描边和阴影信息，系统也未必安装原字体 |
| 完全无痕地擦除原文 | 不可保证 | 动态游戏背景无法仅凭文字框稳定还原；首版应采用遮盖底板，而不是实时图像修复 |
| 性能影响 | 较低到中等 | 文本绘制成本很低，主要新增成本来自布局计算、坐标跟踪和防止 OCR 回读 |
| 实现难度 | 中高 | 需要改造 OCR 坐标链路、翻译任务关联、窗口跟踪和渲染生命周期 |

因此，该需求可以实现“看起来基本覆盖原文”的实用效果，但不应承诺像人工修图一样完全还原背景和字体。VRChat 规则文字、聊天框文字和水平文本最适合此模式；弯曲文字、旋转文字、艺术字和剧烈变化的背景效果会明显下降。

### 12.3 已完成的基础改造

实现保留并扩展了以下基础：

- RapidOCR 输出包含文字多边形框。
- `OcrText` 已保存文字、置信度和坐标框。
- 现有文字组合器已经能根据坐标恢复阅读顺序。
- OCR 识别框已经能够跟随目标窗口，并处理常见 DPI 缩放。
- 项目已有窗口捕获排除能力，可以防止工具窗口进入捕获画面。

本轮已补齐：

1. 取消旧版 1.5 倍外部放大，模型文字框直接保留在捕获帧坐标空间。
2. `OcrText` 同时保留合并文字块、原始行框、画布大小和背景亮度。
3. 翻译请求编号会关联文字布局、目标窗口、识别区域版本和创建时间。
4. 新增与识别区域对齐的独立嵌字层，不复用原有译文浮窗。
5. 翻译完成后校验目标窗口、区域版本和有效期；失效结果不再嵌字。
6. 捕获排除失败时停用嵌字并回退独立浮窗，不使用每帧隐藏/显示方案。

### 12.4 推荐窗口结构

新增独立的“嵌字绘制层”，不要把嵌字直接画进现有 OCR 识别框窗口，也不要复用独立译文浮窗。

推荐窗口职责：

| 窗口 | 职责 |
| --- | --- |
| OCR 识别框 | 调整识别区域、选择单次或持续模式、显示运行状态 |
| OCR 嵌字层 | 只负责原文遮盖和译文绘制；透明、无焦点、鼠标穿透 |
| OCR 独立译文浮窗 | 保留现有列表式译文显示，作为默认模式或嵌字失败时的回退 |

嵌字层的客户区域应与 OCR 识别区域完全一致，不包含识别框的标题栏、按钮和边框。识别框被隐藏时，嵌字层仍可按显示模式继续工作；OCR 停止、目标窗口最小化或目标程序失去前台焦点时，应自动隐藏嵌字层，防止译文覆盖其他软件。

### 12.5 坐标链路

坐标处理必须明确区分以下空间：

1. 目标窗口 Win32 客户区物理坐标。
2. OCR 识别区域物理坐标。
3. 捕获图像实际像素尺寸。
4. OCR 预处理或模型输入尺寸。
5. Qt 绘制使用的逻辑坐标。

推荐保留每一帧的几何变换信息，而不是在各处假定固定的 1.5 倍缩放。OCR 返回后先把多边形框还原到捕获图像坐标，再转换到识别区域物理坐标，最后根据目标屏幕的设备像素比例转换为 Qt 逻辑坐标。

必须覆盖以下测试场景：

- Windows 100%、125%、150%、175% 和 200% 缩放。
- 主屏和副屏缩放比例不同。
- 副屏位于主屏左侧或上方，屏幕坐标为负数。
- 目标窗口最大化、普通窗口、移动和缩放。
- 捕获图像尺寸与目标客户区尺寸不一致。
- 后续模型取消 1.5 倍预处理或改用其他输入尺寸。

### 12.6 原文遮盖方式

只把译文字体画在原文上方通常无法真正盖住原文：译文可能更短，原文字体还可能带有描边、阴影和发光。因此每段译文需要同时绘制遮盖底板。

首版推荐：

- 使用原始 OCR 行框作为遮盖范围，并向四周扩展少量安全边距。
- 对同一翻译块保留所有原始行框，分别遮盖原文，避免用一个过大的矩形挡住无关画面。
- 默认使用高不透明度的中性色圆角底板，并根据文字框周边的内存像素估算明暗，选择深色或浅色底板。
- 译文字体自动选择与底板具有足够对比度的前景色，并保留轻微描边或阴影，以适应亮色和暗色场景。
- 不把用于估算背景的像素写入磁盘；当前帧释放后不保留背景图像。

首版不建议使用图像修复或生成式补背景：这会增加 CPU/GPU 占用、实现复杂度和动态画面伪影，也可能使软件体积显著增加。后续如果确有需要，可以单独评估仅针对小文字框的轻量图像修复，但不能作为本需求的前置条件。

### 12.7 字号与译文排版

字号不能直接使用现有 OCR 译文浮窗的固定字号。推荐流程：

1. 使用原文各行文字框的中位高度估算初始像素字号，降低个别异常框的影响。
2. 使用 Qt 字体度量计算译文在原文字块宽度和高度内的实际占用。
3. 优先保持接近原文的字号，并在原文字块宽度内自动换行。
4. 译文明显更长时逐级缩小字号，但建议不低于估算字号的 65%。
5. 仍无法放下时，优先在识别区域内向下扩展译文块；没有安全空间时才使用更紧凑的排版。
6. 中文和日文使用系统字体回退，不额外打包大型字体文件。

翻译粒度推荐采用“文字块翻译、逐行遮盖”：

- OCR 保留每个原始行框，用于遮盖原文和估算字号。
- 相邻且属于同一段聊天内容的行合并为一个翻译请求，保证翻译上下文和语义质量。
- 译文在合并后的外接区域内重新换行，不要求译文每一行与原文逐字对应。

如果逐行分别调用翻译接口，虽然位置更容易对应，但会增加请求次数、延迟和上下文丢失，不建议作为默认实现。

### 12.8 翻译结果与空间信息关联

每个送入翻译队列的 OCR 文字块必须关联：

- 唯一请求编号。
- 原文文本。
- 原始行框列表和合并后的外接框。
- OCR 识别区域版本。
- 目标窗口句柄。
- 捕获帧签名或布局版本。
- 创建时间和结果有效期。

翻译完成后先校验目标窗口、识别区域版本和有效期。目标窗口已更换、识别框已移动、原文位置明显变化或结果已经过期时，直接丢弃该嵌字结果，但仍可按设置决定是否在独立译文浮窗中显示。

连续识别时使用文字框重叠率和文本相似度跟踪同一段文字，稳定复用已有译文，避免每一帧重新创建绘制项。新结果应批量提交给绘制层并一次刷新，不能频繁创建、销毁或显示、隐藏窗口，以免闪烁和拖动卡顿。

### 12.9 捕获排除与防回读

嵌字层必须调用 Windows 捕获排除能力，使其不会进入 OCR 截图。对于窗口内容捕获，嵌字层与 VRChat 本身是不同窗口，通常不会被捕获；对于屏幕坐标捕获，则必须验证 `WDA_EXCLUDEFROMCAPTURE` 实际生效。

如果当前系统或捕获后端无法可靠排除嵌字层：

- 不允许静默继续持续嵌字，否则译文可能被 OCR 再次识别。
- 不采用每帧隐藏、捕获、再显示的方案，这会产生用户已经遇到过的闪烁窗口。
- 应自动停用嵌字并回退到独立译文浮窗，同时显示明确原因。

### 12.10 UI 与配置

OCR 页面在“译文显示”区域增加显示方式，推荐提供：

- 独立译文浮窗。
- 识别区域嵌字。
- 两者同时显示。

嵌字模式增加最少量设置：

| 设置 | 建议默认值 | 说明 |
| --- | --- | --- |
| 译文显示方式 | 独立译文浮窗 | 保持旧配置和旧使用习惯兼容 |
| 嵌字遮盖强度 | 90% | 确保原文和动态背景不会穿透影响阅读 |
| 自动对比色 | 开启 | 根据背景明暗选择文字和底板颜色 |

字号匹配应始终自动开启，不提供固定字号输入，否则会与“接近原字体大小”的目标冲突。独立译文浮窗仍保留原有字号、透明度、位置和尺寸设置，嵌字设置必须单独分区，避免两种显示方式混淆。

配置升级时新增译文显示模式、遮盖强度和自动对比色字段。旧配置迁移后默认继续使用独立译文浮窗，不能在升级后突然开启嵌字。

### 12.11 性能、存储和隐私影响

| 项目 | 预计影响 |
| --- | --- |
| CPU | 文本布局和绘制开销较低；背景明暗采样应限制在小文字框周边 |
| GPU | 普通 Qt 2D 绘制，增量很低，不引入新的机器学习推理 |
| 内存 | 保存当前有效文字框、译文和少量绘制缓存，通常为数 MB 以内 |
| 打包体积 | 不打包额外字体和图像修复模型时，增量可以忽略 |
| 翻译请求 | 采用文字块翻译时与当前请求数量基本一致 |
| 隐私 | 继续只处理内存帧，不保存截图、背景块或识别历史 |

嵌字功能不会因为 OCR 模型升级而额外增加模型体积。真正影响性能的仍然是 OCR 检测、识别模型和在线翻译延迟。

### 12.12 明确限制

- 无法获知或完全复刻 VRChat 中原文字体的名称、字重、描边、阴影和字距。
- 译文长度与原文差异较大时，字号和换行只能尽量接近，不能保证像素级一致。
- 遮盖底板会挡住文字后方的一小块游戏画面，这是稳定覆盖原文所必须接受的取舍。
- 旋转文字、竖排文字、弧形文字和严重透视文字不纳入首版范围。
- OCR 框错误或把非聊天 UI 识别成文字时，嵌字也会覆盖错误区域，因此仍需要准确限制识别框。
- 在线翻译返回过慢时，原画面可能已经变化；过期结果必须丢弃，不能为了显示结果而覆盖新内容。

## 14. 验收标准

### 功能

- 中文路由只加载中文包。
- 日文路由只加载日文包。
- 缺少模型时不会启动错误的识别器。
- 模型切换后旧识别实例被安全释放。
- 不生成或保存任何截图。
- 用户可以在独立浮窗、识别区域嵌字和同时显示之间切换。
- 嵌字只出现在 OCR 识别框的内容区域，不覆盖识别框外画面。
- 目标窗口最小化、切换或失去前台焦点时，嵌字不会覆盖其他软件。
- 旧配置升级后仍默认使用独立译文浮窗。

### 质量

- 中文小字号识别优于当前 PP-OCRv4 Mobile。
- 平假名、片假名和相似字符识别优于当前日文模型。
- 不依赖大量硬编码字符替换掩盖模型错误。
- 动态背景下的漏识别率和误识别率均有记录。
- 水平中文和日文聊天文字的嵌字位置基本覆盖原文字框。
- 自动字号与原文字框高度接近，较长译文能够在区域内换行或合理缩小。
- 原文字体、描边和阴影不会大面积穿透遮盖底板影响阅读。
- 连续识别时稳定文字不会反复闪烁，过期译文不会覆盖新画面。

### 性能

- 单次 OCR 耗时低于当前 OCR 扫描间隔。
- 连续运行时不出现识别任务堆积。
- OCR 运行时 VRChat 不出现明显持续掉帧。
- 切换语言后内存中只保留当前识别模型。
- 开启嵌字后不出现新的 OCR 任务堆积、持续高 CPU 或 VRChat 明显掉帧。
- 嵌字窗口位置更新和文本刷新不会造成周期性弹窗或拖动卡顿。

### 存储与隐私

- 只存在中文和日文两个可下载识别包。
- 模型与临时文件全部位于软件目录。
- 不使用 C 盘用户缓存作为回退。
- 不保存截图、识别历史或聊天数据库。
- 不额外打包大型字体、图像修复模型或生成式模型。
- 嵌字使用的帧像素和背景采样只存在于内存中，并随当前任务释放。

### 嵌字专项

- 在 100%、125%、150%、175% 和 200% DPI 下，译文框与原文框的主要重叠区域保持正确。
- 在不同缩放比例的多屏环境中移动目标窗口后，嵌字仍能跟随正确位置。
- 嵌字层不能被窗口内容捕获或屏幕坐标捕获再次识别。
- 捕获排除失败时，程序明确回退到独立译文浮窗，不使用每帧隐藏窗口的闪烁方案。
- 翻译超时、识别区域变化或目标窗口切换后，旧嵌字结果不会继续显示。

## 15. 最终建议

当前已经采用：

```text
PP-OCRv5 Mobile Det
+ 中文 PP-OCRv5 Server Rec
+ 日文 PP-OCRv6 Medium Rec
```

检测继续使用 Mobile Det，不使用 Server Det。完成真实 VRChat 中文和日文实测后，再决定是否需要更大的检测模型。

嵌字首版已经按“原始行框遮盖 + 文字块翻译 + 自动字号与换行 + 独立透明绘制层”实现，不引入图像修复模型，也不追求完全复刻原字体。默认仍使用独立译文浮窗，由用户主动选择嵌字或同时显示。
